# 异构 PDF 表格核对流程设计

## 目标

不同监测报告的表头、单位、跨页方式和统计口径会变化。系统不能把“识别出数字”直接等同于“可以硬计算核对”，必须先判断字段是否齐全、单位是否同口径、统计口径是否明确。

## 推荐流程

1. OCR/文本提取
   - 默认先用文本层；文本质量差或扫描件时启用 PaddleOCR 异步 API。
   - OCR 输出保留 raw/clean/debug 三份，便于追溯误报来源。

2. 表格切分与跨页合并
   - 以监测项、测孔编号、表头字段集合、页序作为合并键。
   - 普通表缺测孔编号时按监测项合并；深层位移按监测项 + 测孔编号合并。
   - 续表只补数据行，不重复统计行。

3. 字段语义标准化
   - 将不同叫法映射到标准字段：初始值、本次值、上次累计、本次累计、本次变化、累计变化、变化速率、安全状态。
   - 单位从表头、单位行和数据特征三处共同判断。
   - LLM 可补充判断，但不能覆盖数据证据明确的单位转换规则。

4. 可核对性判定
   - 硬校验：字段齐全、单位明确、公式口径明确。
   - 弱校验：字段基本齐全但基准不可靠、单位需转换、统计口径存在行业惯例。
   - 跳过并说明：缺核心字段、跨页归属不确定、只有汇总无明细、原文 OCR 明显错列。

5. 专项计算器
   - 普通位移：累计变化 = 本次值 - 初始值；变化速率 = 本次变化 / 监测间隔。
   - 高程/沉降：初始/本次通常为 m，变化量通常为 mm，需乘以 1000；若基准不可靠只做 warning/info。
   - 深层位移：按测孔分组；速率可按绝对值比较，跨孔汇总需谨慎。
   - 锚索/支撑：内力类不要求变化速率；累计变化 = 本次内力 - 初始内力。
   - 水位：初始基准和方向约定常变化，容差和严重性降低。

6. 统计与汇总核对
   - 正/负方向最大值要区分方向，不简单按绝对值。
   - 全正或全负时，报告可能填“最小正值/最接近 0 的负值”，应降为 info。
   - 最大速率既可能按绝对值最大，也可能按数值最大；同号速率时需要接受报告口径并说明。

7. 报告输出
   - 明确列出“已硬校验”“弱校验”“跳过原因”。
   - 每个问题标注疑似来源：报告原文、OCR/列映射、规则口径、LLM 解析。

## 速度策略

- OCR 只在需要时启用，结果写入 debug/cache 目录，重复实验复用。
- LLM 只负责结构化和语义补强，公式计算全部由本地规则完成。
- LLM 结构化解析默认按 18000 字符分块、4 路并发；长 PDF 优先通过并发和缓存提速，不通过缩短分块盲目增加请求数。
- 表格分析计划先用规则生成，只有发现单位/基准异常时再调用 LLM 增强。
- 汇总项与分表名称匹配默认使用本地启发式语义映射，避免每份报告额外调用一次 LLM；需要更强语义匹配时可开启 `LOGIC_USE_LLM_SEMANTIC_MATCH=1`。
- 自验证只复核 error 级问题；没有 error 时跳过。
- 默认使用 DeepSeek OpenAI 兼容端点：需要更快响应时使用 `deepseek-v4-flash`；需要更稳结构化时使用 `deepseek-v4-pro`。MiniMax-M2.7 仍作为可选兼容模型保留。

## 2026-05-13 实现更新

- LLM 解析默认分块从 28000 字符收紧到 `LLM_PARSE_CHUNK_CHARS=18000`，并保证单页/单表超过上限时继续二次切分，降低大 PDF JSON 截断风险。
- 多分块结果会去重合并 `thresholds` 和 `summary_items`，避免只保留首块元数据。
- PaddleOCR 缓存键包含 PDF SHA256、Paddle 模型、OCR profile 指纹和清洗器版本；任一变化都会自动失效。
- 当未识别到监测数据表时，逻辑检查新增“数据表识别” warning。此时报告结论为“需复核”，不能把 0 error 视为通过。
- 报告摘要展示 LLM 分块总数、成功数、失败数，并对 Markdown 表格单元格做 `|` 和换行转义。

## 2026-06-08 实现更新

- `LLM_PARSE_MAX_PARALLEL` 默认调为 `4`，桌面端和 Streamlit 均提供配置入口，并同步写入 `RuntimeConfig`。
- Step 5 逻辑匹配默认改为本地启发式映射，恒大中心 99 页 PDF 实测从约 84s 降到 <1s；需要 LLM 语义映射时手动开启环境变量。
- 报告生成器会合并重复的提取质量提示和相同分析计划，长 PDF 不再为每一期重复刷屏。
- 汇总项找不到分表时降为 `info`，表示“该项无法自动一致性核验”，不再制造大批 warning。
- LLM chunk 调用失败或 JSON 解析失败会升级为 `LLM 分块解析` warning，明确说明后续计算只覆盖成功解析出的表。
- 恒大中心无缓存实测显示：基线总耗时 560.12s；快速优化约 305s 但表覆盖不足；平衡复测 469.23s，但 DeepSeek 后段返回 402 余额不足/连接错误，结果必须标注为不完整。

## 2026-06-14 设计修正

业务复核反馈表明，复杂 PDF 的主要风险不是“某个公式没写对”，而是上游表格边界和字段映射不够可审计。后续流程按以下方向调整：

- 新增 Excel 中间层，先让用户看到系统识别出的表格清单、标准化测点、深层位移、统计摘要、问题清单和分析计划。
- 当前 Excel 中间层来自结构化后的 `MonitoringReport`，用于核查最终进入规则引擎的数据；下一阶段会增加 LLM 前候选表格 Excel，用于核查原始 PDF 表格是否漏提、错列或跨页拆断。
- LLM 的目标将逐步从“整份 PDF 到最终 JSON”收窄为“候选表字段语义映射”。这样单次请求更小，失败影响更局部，也便于按表并发。
- 分块并发必须以候选表为最小完整单元，不能为了速度把一张跨页逻辑表切碎。
- 业务文档新增 `docs/表格识别与Excel中间层设计.md`，用于解释表格识别、字段映射、计算核验、不能自动核验的边界和速度优化设计。

## 2026-06-01 实现更新

- CLI 入口 `main.py` 已改为复用 `src.core.pipeline.run_pipeline()`，桌面版、Streamlit、CLI 三个入口共享同一套 8 步流水线，减少行为分叉。
- 跨期累计连续性新增防误报规则：同一日历日（含 AM/PM、上午/下午等不同标签）不做跨期递推；当 `累计_N - 本次_{N+1}` 可解释报告累计时，标记为 info，说明该表本次变化方向约定可能与累计增量相反。
- 宽表统计新增口径识别：若正/负方向最大值与同测点“本次变化”一致，或深层位移最大变化值与同深度“当前累计”一致，不再作为确定 error，而降为 info/warning 供复核。
- 简报汇总一致性降为 warning：该步骤依赖语义映射、监测期选择和汇总口径，不能替代分表硬校验。
- 测试基线不在文档中写死固定数量；以当前 `pytest` 输出、桌面 EXE/MSI 验证脚本、Streamlit 浏览器 smoke 和真实样本回归报告为准。
