# 建筑变形监测报告核验台 · 部署使用说明

> 本文档面向**最终用户（监理工程师、甲方代表、检测公司复核员）**，无需编程背景即可按图操作。

---

## 一、它能帮你做什么

把一份基坑/变形监测报告 PDF 丢进来，几分钟内你会得到一份带颜色标记的检查报告，告诉你：

- 哪些**累计变化量**算错了（公式：本次测值 − 初始测值）
- 哪些**变化速率**算错了
- 哪些"正方向最大""负方向最大""最大速率"的**统计行**与表格里实际极值不符
- 哪些**安全状态**判定与阈值不一致（如累计已超报警值但状态写"正常"）
- 简报汇总表里的数字与各分表是否对得上

支持文字版 PDF 和扫描件 PDF，最终输出 Markdown / Word / 可打印 HTML 三种格式。

---

## 二、系统要求

- **操作系统**: Windows 10 / 11（64 位）
- **内存**: 4 GB 以上（推荐 8 GB）
- **硬盘**: 500 MB 可用空间
- **网络**: 处理过程中需要联网调用 LLM API（DeepSeek / MiniMax / 自定义 OpenAI 兼容端点）

> 整个程序运行在你**本机**，PDF 文件不会上传到任何服务器；只有把表格里的文字片段发送给 LLM 做语义理解，**不包含项目名、报告编号等敏感信息**也可以选择关闭。

## 三、安装与启动

### 方式 A：使用单文件 .exe（推荐给甲方/复核员）

**1.** 把 `BuildingDeformationChecker.exe` 拷贝到任意位置（建议桌面或 `D:\工具\`）。

**2.** 双击启动。首次启动会有 5–10 秒解压时间，看到主窗口即可。

**3.** 在左侧"配置"面板填入：

   | 字段 | 填什么 | 备注 |
   |------|--------|------|
   | **LLM API Key** | 由项目方提供（一般形如 `sk-...`） | 必填 |
   | **Base URL** | 由项目方提供 | 默认 `https://api.deepseek.com` |
   | **模型** | 默认即可 | 自动选 `deepseek-v4-flash`，也可选 `deepseek-v4-pro` |
   | **PaddleOCR Token** | 仅扫描件需要 | 可留空 |

**4.** 点 **保存配置** → 下次启动会自动加载。Key 会被加密存到系统密钥环（Windows Credential Manager），**不会以明文存在文件里**。

如需更换或清除旧 key，点击左侧 **清空已保存密钥**，再重新填写并保存。

### 方式 A2：使用 .msi 安装包（可选）

如果项目方提供的是 `BuildingDeformationChecker-版本号.msi`：

**1.** 双击 `.msi`，按向导安装。

**2.** 从开始菜单或安装目录启动 `Building Deformation Checker`。

**3.** 首次启动同样在左侧配置面板输入自己的大模型 API Key、Base URL、模型名和 PaddleOCR Token。

> `.exe` 和 `.msi` 都不应内置任何真实 API Key。每个使用方第一次输入自己的 key；敏感字段由自己电脑的系统 keyring 保存，后续可自动预填。

### 方式 B：用 Python 源码运行（开发者）

需要预先安装 Python 3.10+：

```powershell
git clone https://github.com/gaaiyun/building-deformation-checker.git
cd building-deformation-checker
pip install -r requirements.txt
copy .env.example .env       # 编辑 .env 填入 API key
python desktop.py
```

开发者如需自行打包：

```powershell
# 生成 dist/BuildingDeformationChecker.exe
powershell -ExecutionPolicy Bypass -File scripts/build_desktop.ps1

# 生成 MSI 需要先安装 .NET 8 SDK + WiX 4
dotnet tool install --tool-path G:\dev-cache\dotnet-tools wix --version 4.0.6
powershell -ExecutionPolicy Bypass -File scripts/build_desktop.ps1 -BuildMsi -Version <版本号> -WixToolPath G:\dev-cache\dotnet-tools\wix.exe
```

## 五、典型耗时

| 报告类型 | 总耗时 | 说明 |
|----------|--------|------|
| 文字版 PDF（20 页内） | 60-180 秒 | pdfplumber 跳过 OCR，主要在 LLM 解析 |
| 文字版 PDF（100 页） | 5-12 分钟 | LLM 分块解析为主 |
| 扫描件 PDF（20 页内） | 4-8 分钟 | 加 OCR 步骤 |
| 扫描件 PDF（100 页） | 15-25 分钟 | 加大量 OCR |

**提速技巧**：
- 跳过 "Step 6 自验证" 和 "Step 7 最终审核" 可以快 30-40%（侧边栏勾选）
- 同一份 PDF 重跑会自动复用 OCR 缓存，第二次很快

## 六、结果如何理解

打开"计算验证"tab 看到：

> ⚠️ **支护结构顶部水平位移** 表
> - 测点 **S3** | 累计变化: 计算应为 42.13mm，报告标 41.83mm，**差 0.30mm**（超过容差 0.15mm）
> - 测点 **S5** | 变化速率: 计算应为 -0.165mm/d，报告标 -0.185mm/d

**红色**=error（明显错误，应人工复核）
**橙色**=warning（疑似，可能 OCR 抖动或边界容差）
**蓝色**=info（提示，如某些字段缺失）

## 七、常见问题

### 1. 提示"未填写 LLM API Key"

→ 左侧配置面板必须填 API Key。点项目负责人要 key，或自行去 [DeepSeek 开放平台](https://api.deepseek.com/) 注册；默认 OpenAI Base URL 为 `https://api.deepseek.com`。

### 2. 运行到 Step 2 卡住 / 超时

→ 可能是 API 服务异常或网络问题。重试一次；如果连续失败：
- 切换模型（侧边栏可在 `deepseek-v4-flash`、`deepseek-v4-pro`、`MiniMax-M2.7` 等 OpenAI 兼容模型间选择，也可以手动输入自定义模型 ID）
- 把 PDF 切成两半分别跑
- 关闭 Step 6/7 复核（侧边栏勾选）

### 3. 扫描件 PDF 提取效果差

→ 配 PaddleOCR Token 后强制 OCR：
- 侧边栏 "PDF 提取方式" 选 "强制 PaddleOCR"
- 首次 OCR 较慢；同一 PDF 第二次会复用缓存

### 4. 报告里有红色错误，但我看着报告又对

→ 工具可能误报（LLM 把表格列读串了）。打开"分析计划"tab 看它认为这是什么类型的表，再看"运行日志"找具体怎么读的。可以反馈给项目方。

### 5. 处理完想换一份 PDF，但按钮变灰

→ 点底部的 **[处理新 PDF]** 按钮即可，不需要关闭程序。

### 6. 程序崩溃 / 闪退

→ 把 `%APPDATA%\BuildingDeformationChecker\logs\desktop_YYYYMMDD.log` 拷贝给项目方排查，并说明 PDF 文件名、LLM Base URL、模型名、是否启用 PaddleOCR。不要发送 API Key。

当前桌面版在关闭窗口时会等待后台 QThread 安全退出，避免“跑到一半主窗被关闭导致闪退”。如果仍出现闪退，优先用源码方式运行 `python desktop.py` 复现一次，这样终端会保留完整 traceback。

## 八、数据与隐私

- **本机处理与外部服务**：默认先用本机 PDF 文本层解析；启用 PaddleOCR API 时，会把 PDF 或页面切片上传到你配置的 OCR 服务；启用 LLM 解析/审核时，会把提取出的文本、表格片段和必要上下文发送到你配置的 OpenAI 兼容模型服务。本工具不会上传到项目方自建服务器。
- **敏感内容处理**：如果 PDF 包含敏感项目名、编号、地址或人员信息，请在上传到外部 OCR/LLM 前先脱敏，或关闭 PaddleOCR/API 审核，仅使用本机文本层和规则核查。
- **API Key 存储**：保存到 Windows 系统的密钥环（Credential Manager），不会以明文落盘
- **清空旧 Key**：桌面版左侧有 **清空已保存密钥** 按钮，适合交付/演示前清掉本机旧 key
- **OCR 缓存**：会在程序目录下的 `output/<报告名>_ocr_debug/` 存 OCR 中间结果，方便复审；不需要时可手动删除

## 十、给项目负责人的部署 checklist

把 `.exe` 或 `.msi` 给甲方/复核员之前，请确认：

- [ ] `.exe` 文件或 `.msi` 安装包本身（不需要 Python 环境）
- [ ] 让甲方通过自己的账号申请/填写 API Key；如确需代配，使用企业认可的密钥管理或加密渠道交付，不要通过便签、普通邮件、zip 或仓库传递真实 key
- [ ] 一份样本 PDF（让对方第一次跑能看到效果）
- [ ] 本文档（或者打印第 4-6 节给非技术用户）
- [ ] 你的微信/电话（万一对方卡住能找到你）

> 你也可以把 `.exe` / `.msi` 和样本 PDF 一起打包成 zip 分发，更友好；不要把真实 key 放进 zip 或仓库。


### 安装包验证

构建 MSI 后可执行：

```powershell
powershell -ExecutionPolicy Bypass -File scripts/verify_msi.ps1
powershell -ExecutionPolicy Bypass -File scripts/verify_msi.ps1 -Install
python -m pytest tests/test_desktop_main_window.py tests/test_streamlit_app_config.py -q
```

第一条只验证文件、大小、哈希和签名状态；第二条会执行 per-user 静默安装、安装后启动、静默卸载冒烟；第三条验证桌面端与 Streamlit 的关键配置和交互出口。

需要隔离真实 API 时，可先启动本地 OpenAI 兼容 mock 服务做 UI 烟测：

```powershell
python scripts/mock_openai_server.py --port 8765
```

然后在桌面版或 Streamlit 中把 Base URL 填为 `http://127.0.0.1:8765/v1`、API Key 任意填、模型填 `smoke-model`。该 mock 只用于验证 UI/后台线程/报告下载闭环，不代表真实 OCR/LLM 识别效果。
