### 3.1 有效结论

- 质安和展誉正确版均为 0 error；对应错误版分别检出 12 和 7 个 error。
- 深工勘错误版检出 10 个 error，包括速率、跨期累计和统计摘要错误。
- 表感知分块把深工勘正确版从逐节约 895 秒降低到约 537 秒，耗时下降约 40%。
- 流式响应解决了长请求在完整响应返回前被中间网络设备断开的风险；结构化结果仍按原 chunk 顺序合并。
- Excel 中间层、Markdown、Word 和 HTML 均随每个样本生成，可逐表复核模型映射。

### 3.2 深工勘正确版的迭代记录

深工勘样本揭示了两个真实问题，不能只看单次 error 数：

1. 末尾“本报告期间最大值”摘要列曾被映射成逐期变化速率，造成正确版误报。
2. 锚索表的“本次(kN)”实际是本期变化，“测值(kN)”才是当前内力；把测值当初始值会产生批量伪错误。

修复包括：结构化表题切块、连续续表合并、摘要表隔离、结果完整性重试、已知力值表头的确定性语义归一化，以及 `initial_value_reliable=false` 时禁止执行初始值差分。历史真实模型请求曾返回 `402 Insufficient Balance`，2026-06-21 当前源码补跑又遇到 600 秒无产物超时；因此最终代码的深工勘正确版结果不能伪写为 0 error。相关规则已有单元测试覆盖，仍需外部服务稳定后做一次无缓存确认。

## 5. PaddleOCR-VL-1.6 实测

2026-06-20 对同一 OCR smoke PDF 做了三层复现：

1. 项目验收脚本强制 OCR。
2. 项目 PaddleOCR runner 直接提交。

三者均在 `https://paddleocr.aistudio-app.com/api/v2/ocr/jobs` 建立连接阶段被远端重置，未得到 HTTP 状态码：

```text

```
