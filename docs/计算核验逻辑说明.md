# 计算核验逻辑说明

本文档说明系统如何定义“可计算核验”、如何处理不同 PDF 模板里的异构表格，以及哪些数据只能提示复核、不能自动判定对错。

对应代码：

- `src/core/pipeline.py`：8 步流水线调度。
- `src/tools/llm_parser.py`：把 PDF 文本结构化为监测表 JSON。
- `src/tools/extraction_quality.py`：判断表格字段是否可信、是否疑似漏行或错列。
- `src/tools/calculation_checker.py`：执行计算公式、跨期连续性、异常值和符号检查。
- `src/tools/statistics_checker.py`：执行正/负方向最大值、最大速率、力值统计核验。
- `src/tools/logic_checker.py`：执行安全状态、汇总表与分表一致性等逻辑核验。

## 0. 架构边界与当前改造重点

系统的最终目标不是“让 AI 读完整份 PDF 后直接给一个结论”，而是把报告拆成可追溯的证据链：

```text
PDF/OCR 文本 -> 候选表格 -> 字段语义映射 -> 标准化监测表 -> 本地规则核验 -> 可审计报告
```

当前代码已经实现了“标准化监测表 -> 本地规则核验 -> 报告”的主体能力，但长 PDF 和复杂跨页表仍存在一个边界风险：`src/tools/llm_parser.py` 会把清洗后的文本按页/字符块拆分，再让 LLM 直接输出最终 `MonitoringReport`。这会把表格边界识别、跨页续表、字段映射、横向多期拆分和业务 JSON 生成压到同一个 LLM 调用里。

因此后续优化的方向是：

- 先建立 Excel 中间层，让业务能核查“系统到底识别了哪些表、哪些字段、哪些测点”；
- 再逐步把 LLM 从“整份报告结构化器”收窄为“候选表字段映射器”；
- 所有计算公式继续由本地 Python 规则执行，LLM 不参与算术判定。

详见：`docs/表格识别与Excel中间层设计.md`。

## 1. 数据标准化后的字段

系统不直接按 PDF 原表头写死规则，而是先把不同公司的表头映射到统一字段：

| 标准字段 | 常见原文表头 | 含义 |
|---|---|---|
| `initial_value` | 初始值、初始高程、初始内力 | 项目首测或报告明确给出的初始基准值 |
| `previous_value` | 上次值、上次测值 | 上一期实测值 |
| `current_value` | 本次值、本次测值、本次高程 | 本期实测值 |
| `current_change` | 本次变化、本期变化 | 本期相对上期或本期内部的变化量 |
| `cumulative_change` | 累计变化、累计位移、累计沉降 | 从初始基准累计到本期的总变化量 |
| `change_rate` | 变化速率、速率、mm/d | 本次变化除以监测间隔天数 |
| `previous_cumulative` | 上次累计 | 深层位移/测斜常见字段 |
| `current_cumulative` | 本次累计 | 深层位移/测斜常见字段 |

单位由 LLM 抽取结果和规则共同决定。常见口径是：高程/水位原始测值为 `m`，变化量为 `mm`，此时公式会乘以 `1000`；水平位移、沉降、裂缝多为 `mm`；锚索和支撑轴力多为 `kN`。

### 1.1 字段映射必须保留原始证据

业务复核时不能只看标准字段名，还要能看到原始表头和来源表。原因是同一个词在不同模板中含义可能不同：

- “变量”可能是 `current_change`，也可能是 `cumulative_change`；
- “本次值”可能是原始高程/坐标，也可能是当前累计值；
- “累计”在深层位移表里常指 `current_cumulative`，在普通位移表里常指 `cumulative_change`；
- “初始值”如果实际是阶段基准或上次值，就不能用来硬核验项目总累计变化。

因此 Excel 中间层会输出标准化后的测点表和问题清单；下一阶段还会增加 LLM 前的候选原始表导出，便于检查跨页续表和错列问题。

## 2. 表格类型画像

系统先判断每张表属于哪类画像，再决定哪些字段必须存在。

1. 绝对值表：有 `initial_value` 或 `current_value`。可核验 `累计变化 = 本次值 - 初始值`。
2. 变化量表/横向多期表：没有初始值和本次实测值，只有 `current_change`、`cumulative_change`、可选 `change_rate`。不把 `initial_value` 缺失视为提取异常。
3. 深层位移/测斜表：按深度行记录 `previous_cumulative`、`current_cumulative`、`current_change` 或 `change_rate`。
4. 力值表：锚索拉力、支撑轴力等，通常不强制要求变化速率。
5. 汇总/统计表：只有最大值、最小值、状态描述，没有逐点明细时，不能做逐点公式核验，只做汇总一致性或提示人工复核。

### 2.1 跨页、跨表和横向多期表

复杂 PDF 最容易出错的不是单个数字，而是“这个数字属于哪张表、哪一天、哪个字段”。系统按以下原则处理：

- 跨页续表：只有当监测项、表头字段、页码连续性和测点序列都支持时，才合并；不确定时输出复核提示。
- 横向多期表：先识别日期列组，再拆成多个单期标准表；不能把整份报告的监测时间段直接当作每个列组的间隔。
- 深层位移表：按测孔编号和深度序列合并，不把不同测孔强行放到同一张连续表。
- 汇总表：只用于统计一致性，不参与逐点公式核验。

当前版本主要在标准化结果层面执行这些规则；后续候选表格中间层会进一步把跨页合并依据暴露给业务审查。

提取质量检查也按画像工作：

- 多期变化量表天然缺 `initial_value`，不会再产生“关键列 initial_value 空值较多”的误报。
- 若一张变化量表已有 `cumulative_change`/`change_rate` 证据，但 `current_change` 大面积缺失，会提示可能漏列。
- 表头测点数和实际解析行数轻微差异不直接报警；差异比例达到约 20% 才作为提取风险提示。

## 3. 普通位移/沉降/水位公式

### 3.1 累计变化

当表中同时有 `initial_value`、`current_value`、`cumulative_change`，且初始基准被判定可靠时：

```text
expected_cumulative = (current_value - initial_value) * unit_conversion
```

其中 `unit_conversion`：

- `m -> mm`：`1000`
- `mm -> mm`：`1`
- `kN -> kN`：`1`

容差：

```text
tolerance = max(table_config.cumulative_tolerance, abs(cumulative_change) * 5%)
```

若超出容差：

- 初始基准可靠：`error`
- 初始基准不可靠或单位/高程口径存在不确定性：通常降为 `warning`

### 3.2 无初始值时的首日基准规则

报告的“监测时间段首日”不一定等于项目首测日。很多周报、日报只把本期起止日期写在封面或概况里，表格里的累计变化仍然是从项目初始基准累计而来。因此系统不能仅凭“表格日期 = 监测时间段首日”就硬判累计应等于本次变化。

只有满足以下任一证据时，系统才把首日当作初始基准日：

- 报告或表格文字出现首测/初测/首次监测/第一期/第 1 次/初始基准/基准日等证据；
- 表内多数测点本身支持该假设，即 `cumulative_change ≈ current_change` 的行数占有效行 60% 以上。

启用该规则后：

```text
first_day_expected_cumulative = current_change
```

也就是首日累计变化应等于本次变化。示例：

```text
监测时间段：2022-05-16 至 2022-05-22
表格日期：2022-05-16
WY1 本次变化 = 0.30
WY1 累计变化 = 0.30
=> 通过
```

如果首日累计不是本次变化，系统输出 `field_name = 首期累计基准` 的 `error`。

该规则的基础条件仍是：

- 表格有明确日期，且等于监测时间段首日；
- 表内没有显式 `initial_value`；
- 测点同时有 `current_change` 和 `cumulative_change`。

如果缺少首测证据，且表内多数行并不支持 `累计≈本次`，系统会跳过该硬校验，只在日志中记录“缺少首测/初始基准证据”。这类场景属于“当前单一 PDF 的字段不足以证明计算口径”的情况，不能制造 error。

### 3.3 变化速率

当有 `current_change` 与 `change_rate` 时：

```text
expected_rate = current_change / interval_days
```

`interval_days` 的选择顺序：

1. 优先使用报告明确给出的监测间隔；
2. 若报告间隔与逐行反推间隔差距不大，采用逐行反推的多数值；
3. 若二者差距明显，用“行级支持率”仲裁，避免多期宽表把整个报告日期范围误当成单期间隔。

如果某个测点反推出的间隔与多数测点不同，但可以解释为点位监测时间不同，系统降为 `warning`；完全不符时为 `error`。

## 4. 跨期累计连续性

横向多期表拆成多张单期表后，系统会按 `monitoring_item + borehole_id` 分组，再按日期排序：

```text
cumulative(N+1) = cumulative(N) + current_change(N+1)
```

这用于没有初始值列的多期模板。系统会跳过以下情况：

- 两张表是同一个日历日，避免同日 AM/PM 或重复抽取被误判为跨期；
- 后一期是前一期的完整值复制，疑似 LLM 重复抽取；
- 测点编号无法匹配。

如果按相反符号可以连续，系统输出 `info`，说明该模板的“本次变化”方向约定可能与累计增量相反；否则输出 `error`。

## 5. 深层位移/测斜公式

深层位移表按深度行核验：

```text
current_change = current_cumulative - previous_cumulative
change_rate = current_change / interval_days
```

特点：

- 没有 `change_rate` 列时不强行检查速率；
- 若只有 `previous_cumulative + current_cumulative + change_rate`，可以用两累计值推导本期变化；
- 速率比较允许按绝对值判断，因为部分测斜表的方向符号约定与普通位移不同；
- 统计块不存在时，不把最后一行数据当统计。

## 6. 锚索/支撑等力值表

内力类表主要核验：

```text
cumulative_change = current_value - initial_value
```

力值表通常不要求 `change_rate`。当模板只有当前内力、最大/最小内力等统计数据，而没有初始内力或累计变化时，系统不能做公式核验，只能检查统计摘要或提示人工复核。

## 7. 统计核验

统计核验不只取绝对值最大，而是尊重方向：

- 正方向最大：正值中数值最大的点；
- 负方向最大：负值中绝对值最大的点，也就是数值最小的点；
- 最大变化速率：一般按绝对值最大，但报告模板若明确采用“正向最大/负向最大”口径，会降为复核提示；
- 力值表：检查最大内力、最小内力、对应测点。

如果汇总表只有文字描述或缺测点明细，系统不会硬判对错。

## 8. 异常值与符号一致性

除了公式，系统还会做两个防漏检查：

1. 单期变化幅度异常：
   - `abs(current_change)` 远大于同表测点中位数；
   - 或 `abs(current_change)` 明显大于 `abs(cumulative_change)`。
   - 输出通常为 `warning`，提示核对原 PDF。

2. 符号一致性：
   - 若 `current_value - initial_value` 与 `cumulative_change` 符号相反，且量级接近，输出问题；
   - 如果两者量级差距超过 10 倍，系统认为 initial 列可能不是项目首测基准，跳过，避免误报。

## 9. 不能自动核验的情况

以下数据只能提示“无法核验”或“需人工复核”，不能自动判定正确/错误：

- 单一 PDF 表格只有 `current_change`，没有 `cumulative_change`、`previous_value`、`current_value` 或日期间隔；
- 只有汇总统计，没有逐点明细；
- 表头缺单位，且数值无法通过数据分布可靠判断单位；
- 初始值列实际含义可能是“上次值”或“某一阶段基准”，但报告没有说明；
- 跨页表格测点编号被截断，无法确认两页是否同一张表；
- OCR 输出存在明显损毁，如长串重复字符、整页重复行；
- LLM 结构化解析存在失败分块，导致部分页面没有进入后续规则引擎；
- 安全状态只写“正常/稳定”，没有阈值或报警值来源。

系统在这些场景下应输出 `info` 或 `warning`，并说明跳过原因，而不是制造 `error`。

## 10. 速度优化策略

LLM 只负责结构化和语义判断，所有公式计算都在本地 Python 完成。当前速度相关策略：

- Step 2 LLM 结构化解析支持分块并发，环境变量：

```powershell
$env:LLM_PARSE_MAX_PARALLEL = "3"
```

默认并发数为 `4`。如果服务端限流或余额不足，可以降为 `1` 或 `2`；如果接口稳定且速率额度足够，可保持 `4`。

- 分块结果按原 chunk 顺序合并，避免并发导致报告元数据和表格顺序错乱。
- 如果某些 chunk 调用失败或 JSON 解析失败，报告会生成 `LLM 分块解析` warning，后续计算只代表成功解析出的表格，不能视为完整通过。
- LLM 响应缓存按请求参数和消息哈希命中，重复实验时不会重复付费请求。
- 逻辑检查的名称映射默认走本地启发式规则，避免每份报告额外等待一次 LLM；需要更强语义匹配时设置 `LOGIC_USE_LLM_SEMANTIC_MATCH=1`。
- Step 6 自验证只复核 `error` 级问题；无 error 时可跳过。
- Step 7 最终审核可关闭，批量测试和性能回归建议关闭。
- PaddleOCR 结果按 PDF 哈希、模型和 OCR profile 缓存；正式从头测试时需显式关闭或清理缓存。

### 10.1 为什么不能只靠提示词提速

如果一个 LLM 请求同时承担以下任务，请求会天然变慢且更容易失败：

1. 判断哪些文本是表格；
2. 判断跨页续表；
3. 映射字段语义；
4. 拆分横向多期；
5. 生成最终业务 JSON；
6. 避免漏表漏行。

这不是单条提示词能完全解决的问题。更稳的做法是把任务拆细：

- 本地规则先做候选表格清点；
- Excel 中间层输出候选表和标准化表；
- LLM 只处理每张表的字段语义，且可并发；
- 计算和统计全部由本地规则执行；
- 对深工勘、展誉、质安等已知模板逐步沉淀 fast path。

### 10.2 并发的边界

并发能减少等待时间，但不能牺牲表格完整性：

- 可以并发处理不同候选表；
- 可以并发处理不同 PDF；
- 不应把一张跨页逻辑表粗暴切成多个互相不知道上下文的 LLM 任务；
- 合并结果时必须按页码、表格编号和日期排序，不能按接口返回顺序排序。

## 11. 问题严重度

| 等级 | 含义 |
|---|---|
| `error` | 公式或统计在当前字段口径下明确不成立，应优先核对 |
| `warning` | 可疑问题、接近阈值、字段/单位/基准存在不确定性 |
| `info` | 规则边界说明、跳过原因、可能的模板口径差异 |

报告中每个问题还会尽量标注疑似来源：报告原文、OCR/列映射、规则边界或 LLM 解析。
