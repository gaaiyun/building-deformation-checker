## 工具的"正确行为"应该是
- 不应识别为有监测项目数据：
  - 无监测项目数据 → 没有真实的报警值/控制值阈值配置（仅有 `FLOAT_TOLERANCE / RATE_TOLERANCE` 等代码常量）。
  - 无具体测点行 → 无法扫出"测点编号 + 初始值 + 本次值 + 累计变化"四元组以上的数据行（正则命中 = 0）。
  - 无监测日期 / 监测单位 / 报告编号 / 监测期次等典型监测报告元数据字段。
- 应在最终报告里明确说明"**识别失败：未检出监测数据表**"或类似 negative result，而不是把"动态验证配置表 / 检查规则详解表 / 技术栈表"等**软件规则配置表**错误识别成监测数据表后编造测点。
- **error 数 = 0**（本来就不是监测报告，不存在"计算错误 / 统计错误 / 安全状态错误"）。
- **warning 数 = 1**（一条"未识别为监测报告 / 未检出监测数据表"的元层提醒；如工具进一步识别出文档类型为 README/设计说明，建议升级为更明确的 info "文档类型非监测报告，已跳过验证"）。
- **状态：N/A** 或 "**非监测报告**"。

## 关键测试点
- 工具是否会把"动态验证配置表 / 检查规则详解表 / 支持的监测项对照表"等**软件规则配置表**误识别为监测数据表？
  - 风险点：这些表里出现了 `mm / m / kN / cumulative_tolerance / severity / 累计变化 / 速率 / 报警 / 控制值` 等监测领域术语，LLM 极容易被"骗"成监测表 schema。
- 工具是否会把章节里的数字（如 "3.5 节"、"1. PDF 数据提取"、"步骤 1~7"）误识别为测点编号？
  - 风险点：章节号 "5.3.4" 与某些监测公司的测点编码格式（如 "5-3-4"）相似度高。
- 工具是否会把"高程数据特殊处理"段落里的**举例数值** "初始高程=-2.70184 / 本次高程=-2.70242 / 累计变化=31.21" 抽成一行真实测点数据？
  - 风险点：这是一处文档里**用来举反例**的教学性数据，正常监测表识别逻辑无法判断这是举例还是真实数据；理想情况下 LLM 应该靠上下文（"原因: 报告中的'初始高程'是本期的参考值"等说明性文字）判定为非数据行。
- 工具是否会把"测试记录"章节里的统计描述（"数据表: 12 张 / 错误: 8 个 / 警告: 26 个"，以及 "177 个深层位移速率错误 / 82 个跨表引用误报"）当成本份报告自身的检查结果？
  - 风险点：这是项目历史回归测试记录，但表述形式与"最终结论"很像，可能污染最终结论生成。
- 工具是否会把 `requirements.txt` 里的 `openai>=1.0.0 / pdfplumber>=0.11.0` 这类版本号当成测点测值？
  - 风险点低，但若 LLM 上下文窗口对齐错位仍有可能。
